# I Built My Own LLM Completely From Scratch (here's how you can too)

Video ID: `E4DhRA819Xo`

## Summary
This video walks through building a small language model from scratch on a consumer laptop, covering all four stages: tokenization, architecture, pre-training, and supervised fine-tuning. The creator trains a ~30–32 million parameter model on transcripts of their own YouTube videos plus a generic dataset from Hugging Face (Cosmopedia), using Claude to write and run the Python scripts rather than coding everything manually. The finished model can answer questions about topics it was trained on (e.g., "what is vibe coding?") but fails completely on out-of-domain questions. The core argument is that building an LLM is accessible and nearly free, but the result is far too limited for real business use — the video is most relevant to developers and curious non-engineers who want a hands-on mental model of how LLMs work.

## Key insights
- **LLMs are next-word predictors.** Every response is generated by repeatedly asking "what is the most likely next token?" given the training data.
- **Training data used:** (1) transcripts of the creator's own YouTube videos, (2) Hugging Face's Cosmopedia dataset (textbook-style generic English). Both were needed because the YouTube data alone was insufficient.
- **Tokenization converts words to numbers.** Each word (or sub-word fragment) maps to a unique integer token; common words get the same ID every time, enabling efficient processing. Output is a `tokenizer.json` file.
- **Model architecture = the "brain infrastructure."** The model here uses 512 nodes per layer across multiple layers. Parameters are the numeric weights on connections between nodes.
- **Pre-training = adjusting weights through repeated passes.** The model predicts the next token, compares output to ground truth, measures the gap (loss), nudges weights ("knobs"), and repeats. Loss steadily decreased across 2,500+ steps over ~1.5 hours on a 2025 MacBook Pro with 32 GB RAM.
- **Parameter count of the demo model: ~30–32 million.** For comparison: Gemma 4 (E4B) has ~4.5 billion (roughly 150× more); Fable 5 is estimated at ~6 trillion (roughly 500,000× more).
- **After pre-training only, the model speaks English but can't hold a coherent conversation.** It produces grammatically plausible but semantically nonsensical output when asked "hello."
- **Supervised fine-tuning (SFT) teaches the model how to behave.** AI-generated question-answer pairs (derived from the YouTube transcripts) were used as fine-tuning examples. SFT runs much faster than pre-training.
- **Post-SFT results on in-domain questions were surprisingly usable:** "What is vibe coding?" → "Describing what you want in plain language and letting AI handle the details." "Biggest mistake with AI?" → "Chasing new tools before figuring out the actual problem."
- **Out-of-domain questions completely break the model.** Asking "what is football?" produced incoherent output, illustrating how small training data + small parameter count = no generalization.
- **The whole build cost essentially $0 and ~2–3 hours of wall-clock time** (mostly waiting for training runs), using only Claude to orchestrate pre-written Python scripts.
- **Real frontier models cost billions of dollars and months of compute** — the gap between a hobbyist model and production models is ~500,000× in parameters alone.

## Use cases
- **Developers or technical learners** who want an intuitive, hands-on understanding of how LLMs work without needing a research background.
- **Educators or students** looking for a concrete demonstration of tokenization, pre-training, and fine-tuning as distinct stages.
- **Anyone building a niche domain assistant** (e.g., a "digital twin" chatbot trained on their own content) who wants to understand what's possible and where the limits are.
- **Content creators** curious whether their video/blog corpus can be turned into a personalized assistant (short answer: yes, but only for in-domain questions).
- **Business owners evaluating AI options** — the video explicitly argues that building your own LLM is *not* the right move for real business applications; using existing models and knowing where to apply them is more valuable.

## Patterns & frameworks

**4-Stage LLM Build Pipeline**
A repeatable process for constructing a custom LLM:
1. *Tokenization* — convert raw text to integer token IDs, producing a vocabulary file (`tokenizer.json`).
2. *Architecture* — define the model structure (number of layers, nodes per layer, embedding dimensions); this sets the parameter count ceiling.
3. *Pre-training* — run many forward passes through the architecture, comparing predicted tokens to ground truth, and back-propagating to adjust weights until loss converges.
4. *Supervised Fine-Tuning (SFT)* — feed the model curated question-answer pairs to teach response style and behavior, not just language fluency.

**Loss-as-progress signal**
Training loss is tracked at every step and used as the single metric for model improvement. Watching loss decrease over 2,500 steps provides a concrete feedback loop without needing to evaluate outputs manually at each step.

**AI-assisted SFT data generation**
Rather than hand-authoring fine-tuning pairs, the creator used an AI (Claude) to read the raw corpus and synthesize Q&A examples automatically. This is a reusable pattern: feed source content → prompt AI to extract likely questions and ideal answers → use output as SFT pairs (with optional human review).

**"Knobs" mental model for weight updates**
Weights are described as knobs that get nudged slightly after each pass — a simple but accurate intuition for gradient descent without requiring mathematical detail.

**Parameter count as capability proxy**
The video uses parameter count as a rough but practical stand-in for model quality, enabling quick comparisons: hobbyist model (32M) vs. Gemma 4 E4B (4.5B) vs. Fable 5 (~6T).
